# AI 总体架构在 SeaPack 项目中的落地规划

## 一、现状与差距分析

### 1.1 当前项目定位
当前 `seapack-template` 是一个基于 **Vue 3.5 + TypeScript + Vite 5** 的企业级后台管理系统模板，已具备：
- 前端工程化能力：Element Plus、Pinia、Vue Router、Axios、UnoCSS、ECharts、Cesium、AntV X6
- 业务模块：系统管理（用户/角色/部门/菜单/字典）、股票分红管理、股票行情监控、基金信息管理、用户股票监控池
- AI 前端接口：流式对话（`/api/chat/aiModel`）、Agent 调用（`/api/agent/run-agent`）、RAG 知识库（`/api/rag/*`）、文生图（`/api/ai/*`）
- 权限模型：基于 RBAC 的菜单与按钮级权限

### 1.2 与目标架构的核心差距
| 维度 | 当前状态 | 目标状态 |
|------|---------|---------|
| 架构层级 | 前端 + 零散接口 | 四层完整 AI 中台 |
| 后端能力 | 无完整后端或仅有 Mock | 微服务集群 |
| 模型接入 | 单点直连 | 模型网关统一调度 |
| 知识库 | 简单文件索引 | 企业级 RAG 与向量库 |
| Agent | 单次调用 | 多轮、工具调用、工作流编排 |
| 可观测 | 基本日志 | 全链路监控、评测、治理 |
| 业务场景 | 通用后台 + 股票基金数据管理 | AI 驱动的投研、投顾、风控、投教 |

## 二、四层架构映射（股票基金场景版）

### 2.1 基础设施层
为整个 AI 中台提供算力、存储、运行环境与基础中间件。

| 组件 | 股票基金场景下的作用 |
|------|---------------------|
| 云资源 / GPU 算力 | 运行私有化大模型（如 LLaMA、ChatGLM）、Embedding 模型、文生图模型 |
| 集群服务 | Kubernetes 集群，承载所有微服务 |
| 存储库 | 向量数据库（Milvus/PGVector）存储研报/公告/法规的 Embedding |
| 数据库 | PostgreSQL/MySQL 存储业务数据；时序库（TDengine/InfluxDB）存储行情数据 |
| 对象存储 | MinIO/OSS 存储研报 PDF、图片、模型产物、工作流快照 |
| 消息队列 | RabbitMQ/RocketMQ/Kafka 处理行情流、异步任务、预警通知 |
| 容器与运行环境 | Docker + K8s + Istio，支持 GPU 调度（NVIDIA Device Plugin） |
| 网络与安全 | Nginx/Kong、WAF、TLS、mTLS 服务网格、IAM 统一认证 |

### 2.2 AI 平台层
沉淀统一、可复用、可治理的通用 AI 能力。

| 模块 | 功能说明 | 股票基金场景示例 |
|------|---------|-----------------|
| 模型网关 | 统一接入多厂商/私有化模型，负载均衡、限流、降级、计费 | 同时接入 GPT-4、通义千问、本地 LLaMA-70B |
| 知识库与 RAG | 文档解析、切分、Embedding、检索、重排序 | 研报/公告/财报/法规问答 |
| Agent Runtime | 智能体运行时，支持 ReAct / Plan-and-Execute | 投顾 Agent、投研分析 Agent |
| Workflow Runtime | 可视化工作流引擎 | 基金诊断流程：数据获取 → 指标计算 → 模型分析 → 报告生成 |
| Tool / Skill | 工具与技能市场 | 股票行情查询、分红计算、财务指标计算、舆情抓取 |
| Sandbox | 代码执行沙箱 | Python 量化脚本回测、指标计算 |
| 任务中心 | 异步任务调度与执行 | 批量研报解析、定时行情监控 |
| 文件中心 | 文件上传、解析、版本管理 | 研报 PDF 解析为 Markdown/Chunks |
| 观测与治理 | 日志、监控、链路追踪、模型评测 | 回答准确率评估、调用耗时监控 |
| 开放 API / SDK | 对外暴露 REST/SSE/WebSocket/SDK | 供 APP、小程序、第三方系统调用 |
| 权限与资源授权 | RBAC + ABAC，API/模型/算力配额 | 按角色分配模型调用额度 |

### 2.3 AI 专家层
将平台通用能力封装为面向业务的专业 AI 场景。

| 模块 | 功能说明 |
|------|---------|
| AI 专家门户 | 领域专家（投研、风控、合规）管理自己的知识库与场景 |
| 场景管理 | 定义股票分析、基金诊断、风险预警等场景 |
| 专家工作台 | Prompt 工程、知识库标注、工具编排、工作流设计 |
| 智能 / HITL | 人机协同：模型输出审核、答案标注、反馈闭环 |
| 专家模板 | 可复用的场景模板库（股息分析模板、基金对比模板） |
| 发布与配置 | 场景版本管理、灰度发布、A/B Test |
| 任务协同 | 专家标注任务分配、审核流程 |

### 2.4 AI 应用层
面向最终业务用户与业务系统落地，聚焦股票基金方向。

| 应用场景 | 功能描述 |
|---------|---------|
| AI+股票分析 | 个股诊断、技术面/基本面分析、舆情情绪分析、相似股推荐 |
| AI+基金诊断 | 基金持仓分析、基金经理画像、风险收益评估、基金对比 |
| AI+股息监控 | 智能分红预测、股息率预警、分红方案解读、历史分红分析 |
| AI+行情预警 | 基于阈值 + AI 模型的异动预警、趋势预测 |
| AI+投研报告 | 研报摘要、行业对比、事件影响分析、投资建议生成 |
| AI+风控合规 | 持仓风险扫描、合规审查、异常交易识别 |
| AI+投教客服 | 投资知识问答、新手引导、术语解释 |
| 研发效能平台 | 代码生成、测试用例生成、接口文档生成、SQL 生成 |

## 三、硬件配套方案

### 3.1 MVP / 开发环境（最小可用）
| 用途 | 配置建议 | 数量 |
|------|---------|------|
| 应用服务器 | 16C / 64G / 500GB SSD | 2 台 |
| GPU 服务器 | 单卡 A10 24G 显存 或 RTX 4090 24G | 1 台 |
| 数据库服务器 | 16C / 64G / 1TB SSD | 1 台 |
| 对象存储 | MinIO 单节点 或 云 OSS | 1 套 |
| 网络 | 千兆内网、固定公网 IP | - |

### 3.2 生产环境（高可用）
| 用途 | 配置建议 | 数量 |
|------|---------|------|
| K8s Master | 8C / 32G / 200GB SSD | 3 台 |
| K8s Worker（CPU） | 32C / 128G / 1TB SSD | 5+ 台 |
| GPU 节点 | 2-4 卡 A100/H100 或 8 卡 RTX 4090 | 2+ 台 |
| 数据库集群 | PostgreSQL 主从 + Milvus 集群 | 3+ 节点 |
| 时序数据库 | TDengine / InfluxDB 集群 | 3 节点 |
| 对象存储 | MinIO 集群 或 云厂商 OSS | 高可用 |
| 负载均衡 | Nginx / 云 SLB / Kong | 2 台 |
| 全链路可观测 | ELK + Prometheus + Grafana + Jaeger | - |

### 3.3 网络与安全
- 内外网隔离：DMZ 区部署网关，内网部署核心服务
- TLS 1.3 全链路加密
- 服务网格 mTLS（Istio）
- WAF + DDoS 防护
- VPN / 堡垒机运维接入
- 数据分级：公开数据、内部数据、敏感数据（用户持仓、交易记录）加密存储

## 八、风险与建议

| 风险 | 建议 |
|------|------|
| 大模型私有化成本高昂 | 初期可优先使用云 API，后期再引入私有化模型 |
| 金融数据合规敏感 | 用户持仓、交易数据必须脱敏/加密，本地化部署优先 |
| 模型幻觉影响投资决策 | 关键结论必须引用数据源，引入 HITL 审核 |
| 业务场景过多导致战线过长 | 优先落地 1-2 个高频场景（如智能投顾、股息监控） |
| 后端从零建设周期长 | 可考虑引入成熟 AI 中台框架（如 Dify / FastGPT / RAGFlow）作为起点 |
| 团队技术栈跨度大 | 建议组建跨职能团队：后端、AI 工程、前端、业务专家 |

---

## 九、下一步建议

由于你当前项目是一个前端模板，若希望尽快验证价值，建议先做 **Phase 1 的最小闭环**：
1. 部署一套开源 AI 中台（如 Dify / RAGFlow）作为后端。
2. 将现有 `src/api/ai/*` 接口对接到该平台。
3. 优先实现 **AI+股票分析** 或 **AI+股息监控** 一个场景。
4. 跑通后再逐步扩展为完整四层架构。

这样可以在 2-4 周内看到第一个可用版本，降低全栈建设的风险。
